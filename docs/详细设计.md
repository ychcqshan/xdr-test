# XDR 平台详细设计文档

本设计文档旨在系统性地梳理本项目（云管端混合架构的 XDR - 扩展检测与响应平台）的核心业务逻辑、技术数据流向和全栈模块规范。包含端点采集探测（Agent）、云原生微服务（Backend）和数据可视化视图（Frontend）。

---

## 🛡️ 一、 探针端点端（Agent 端）功能与业务逻辑

XDR Agent 驻留在用户终端与服务器节点，是风险感知系统的前哨。其核心任务是**持续获取终端上下文、发现异动事件、实施主被动响应隔离。**

### 1.1 功能清单与采集逻辑

| 功能模块 | 采集的数据与维度 | 采集方式与技术原理 | 上报模式与机制 |
| :--- | :--- | :--- | :--- |
| **基础维度信息 (Host)** | 主机名、系统版本、CPU/内存状态、IP与MAC | 利用 `psutil` 及 `socket` 实时侦听 | 心跳 (`HEARTBEAT`) 每 5 分钟全量上报 |
| **实时进程监控 (Process)** | PID、进程名、启动时间、命令行路径、占用率 | 利用 `psutil.process_iter()` 遍历 | 动态识别基线变化，采用增量/全量 (`INCREMENTAL`/`FULL`) |
| **网络端口侦测 (Network)** | 协议(TCP/UDP)、绑定本地/远端 IP 端口、所属进程 PID | `psutil.net_connections()` 抓取封包状态 | 增量上报连接变化 (`INCREMENTAL`) |
| **操作系统登录审计 (Login)** | 登录用户、终端源 IP、登录时间 | 查询系统审计日志(Linux `wtmp` / Win `EventLog`) | 完整覆盖式全量上报 (`EXTRA` / `LOGIN`) |
| **外部媒介设备 (USB)** | U盘挂载事件、设备厂商与标识参数 | 查询总线信息及注册表变更插拔事件 | 完整覆盖式全量上报 (`EXTRA` / `USB`) |
| **全网流量截获 (Traffic)** | 源/目的 IP 地址、协议类型及连接频次 | `scapy` 网卡双向混杂模式抓包与统计 | 作为 `TRAFFIC` 独立事件按周期全量或增量上报 |
| **勒索软件探针 (Ransomware)** | 暗哨诱饵文件被异常修改加密的事件 | `watchdog` 结合 FileSystem 操作系统内核监听 | 实时发现，上报高危 `ALERT` 告警 |

### 1.2 Agent 本地数据缓存与上报机制
1. **增量对比与动作码 (Incremental Logic)**:
   - Agent 将上一周期的快照指纹（如 `pid|name|createTime`）存于内存。
   - 对比发现差异后，上报带有 `action` (`ADD`/`REMOVE`/`UPDATE`) 的增量包。
   - **全量兜底 (`FULL`)**：Agent 启动或按长周期触发，用于校准云端状态。
2. **中心云聚合状态机 (Cloud State Machine)**：
   - **状态终态化**：`asset-service` 维护 `host_asset_record` 表，通过 `status` (`ACTIVE`/`INACTIVE`) 标记存量。
   - **逻辑流转**：接收到 `ADD` 则 Upsert 并激活；接收到 `REMOVE` 则逻辑下线；`FULL` 模式下通过快照指纹比对实现存量资产的一致性同步。
   - **前端呈现**：默认仅透视 `ACTIVE` 记录。
- **UI/UX 优化规范**：详情见 `docs/web_design.md`（包含 Popover 悬浮、13px 标签标准、详情页前端分页等）。
3. **扩展字段过滤逻辑 (Expansion Filters)**：
   - 后端 `AssetService` / `AlertService` / `BaselineService` 均已实装对 `unit` (单位) 和 `responsiblePerson` (责任人) 的检索能力。
   - 业务逻辑采用 `query.eq` 或 `query.like`（取决于具体实现）确保管理人员可以按组织架构维度进行态势穿透。
2. **长连接通信**:
   - 上报路由：将最终的事件 JSON 交由 `CommManager` 模块，向后端的 `api-gateway` (`http://localhost:8080/api/v1/events`) 抛出 `POST` 请求投递。
   - 命令执行：10秒一个轮回地拉取待处理的 `PENDING` 策略命令（如阻断网络、杀除 PID 进程），并在终端下发系统函数销毁后将结果 `EXECUTED` 退回云端。

## ☁️ 二、 云端微服务（Backend 端）解析与存储逻辑

平台依托 Spring Boot 微服务集群构建，共提供横向扩展的模块以消化和分析海量采集端点数据。

### 2.1 微服务模块清单
- **Gateway (`api-gateway`)**: 作为单点登录检验（JWT校验）与前置路由过滤。
- **Threat Service (`threat-service`)**: **核心服务。** 专门用于洗流、存储终端资产指纹快照，并作告警关联。
- **Policy Service (`policy-service`)**: 用以记录向 Agent 下发的防守指令与隔离操作。
- **Asset Service (`asset-service`)**: 以主机、组织维度建立管理，负责拓扑统计与视图聚合展示汇出。
- **Auth Service (`auth-service`)**: 处理云端控制台用户与端点 Agent 的准入 Token 分发机制。

### 2.2 数据接收、解析与存储逻辑
1. **入口接纳 (`EventController`)**:
   - Agent 请求抵达 `threat-service` 的 `/api/v1/events`。
   - 原始 JSON 落至 `xdr_threat.event`，保留原始审计日志。
2. **解析与跨服务同步 (`AlertService`)**:
   - `threat-service` 异步解析 eventData，识别资产指纹后通过 `RestTemplate` 将解析后的数据转发至 `asset-service` 的 `/api/v1/assets/internal/sync`。
3. **时序聚合存储 (`HostAssetRecordService`)**:
   - `asset-service` 接收同步包（`SyncAssetRequest`），根据 items 中的 `action` (`ADD/REMOVE/UPDATE`) 驱动状态机。
   - **指纹聚合**：在 `host_asset_record` 中进行特征指纹 (Fingerprint) 的 Upsert 或 Inactive 操作。
   - **解耦优势**：实现了安全威胁分析与基础资产生命周期管理的业务解耦。

## 🖥️ 三、 管理前端（Frontend 端）交互与状态呈现

前端基于 Vue 3 采用 `Fintech Elite Pro Max` 的重装皮肤渲染海量复杂的态势信息。

### 3.1 核心功能概览
- **安全仪表盘 (`DashboardView`)**: 
  - 全球威胁大盘、风险资产占比饼状图、以及滚动的情报流瀑布。
  - 通过 ECharts 定时局部数据拉取策略形成 Mesh 渐变视差动态面板。
- **资产管控中心 (`AssetListView` / `AssetDetailView`)**: 
  - 通过列表透视所有终端与 Agent 连接心跳监控状态。
  - **资产聚合视图**：在详情页中，直接横向调取后端组合 API `getAssetDetails`。此 API 聚合了基础配置并在其底层向 `threat-service` 调用微服务查询获得终端详尽的 `PROCESS` (进程列表)、`LOGIN` (登录历史) 等实况渲染于 Tabs。
- **威胁告警大厅 (`AlertListView`)**: 
  - 获取由勒索规则与基线检测反馈的异常事件栈，并借此向 `PolicyService` 打出拦截处突命令下发请求。

### 3.2 读取与视图刷新
- 前端对后端通信封装于 `src/api`，携带 Bearer Token 实现鉴权。
- **组件动画刷新**：列表与展示卡片深度利用 `entrance-stagger` 和 CSS `delay` 实现错落交响的渐入，不仅能掩盖请求拉取的 Latency，还在呈现前提供极其柔顺的用户体验支撑。数据抵达之后触发 Vue 响应式依赖系统立刻完成试图同步。

## 📝 审计提示：修复一致性设计盲点（数据进程及日志空缺）
经由全栈排查，关于**前文反馈的：“仪表盘终端采集数据进程、登录日志列表皆为空” 一致性 Bug**，已得到彻底清源修复：
**【缺陷定位】**：
在 Phase 2 从单体改造向跨服务组合聚合数据的过程中，由于后端 `threat-service` 在 `AlertService` 中引用了 `RestTemplate` 组件跨域调用数据，但微服务遗漏了向 Spring 注入原生 `RestTemplate` 的 Bean，从而引发 `threat-service` 在首次初始化时触发致命错误并在启动周期陷入 `UnsatisfiedDependencyException` 完全宕机崩溃。
这最终使得外部的 Gateway 将由于解析错误反弹给 Agent `500 Internal Error` 以及 API调用方 `401 / 500` 错误请求状态异常——**故进程和登录快照数据无法被正常同步入库。**
**【整改执行】**：
目前均已在源码工程中于各自的微服务中补充了 `@Bean RestTemplateConfig`。经过整体 `mvn install` 和重新拉起应用，数据采集长效链条恢复互通，资产明细页内进程及登陆项数据现已可顺利读写！

## 🚀 四、 路线图演进：Phase 6 资产管理模型升维 (Time-Series)

为了从“离散快照”走向“全生命周期追踪”，项目正处于向 Phase 6 演进的交汇点。

### 4.1 核心机制转变
- **从覆写到追踪**：废弃 `DELETE-THEN-INSERT` 的逻辑。引入 `host_asset_record` 模型。
- **状态流转算法**：
    - **首次触达 (First Seen)**：新特征出现时记录起点。
    - **持续保活 (Last Updated)**：心跳或数据同步时刷新终点。
    - **自动消亡 (Inactive)**：若本次全量同步未见某特征，将其标记为 `INACTIVE`。
- **时光机查询**：前端将支持回溯历史任意时刻的进程/端口状态快照。

### 4.2 架构职责迁移
- **归还领域主人**：原本暂存于 `threat-service` 的资产同步解析逻辑将完全迁移至 `asset-service`。
- **存储优化**：`threat-service` 仅保留 `event` 审计流水，`asset-service` 维护最终的 `host_asset_record` 状态库。
