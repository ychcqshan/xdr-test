# 🎮 Mode Switching (核心机制)
用户可以在任务开始时通过指令指定开发模式。**默认模式为 `/pdd`**。

| 模式指令 | 名称 | 适用场景 | 核心行为 |
| :--- | :--- | :--- | :--- |
| **`/pdd`** (Default) | **Prompt-Driven** | 复杂业务逻辑、新模块设计 | **先逻辑后代码**。输出详细逻辑计划/伪代码 -> 用户确认 -> 生成代码。**不强制写测试**。 |
| **`/tdd`** | **Test-Driven** | 核心算法、金融/支付逻辑、公共库 | **先测试后代码**。先生成完整单元测试 -> 用户确认 -> 生成实现代码 -> 确保测试通过。 |
| **`/cbd`** | **Contract-Based** | API 开发、数据结构、前后端对接 | **先定义后实现**。先定义 Interface/Type/Schema -> 用户确认 -> 基于类型严格实现。 |
| **`/fast`** | **Rapid Prototyping** | 简单功能、UI 调整、脚本、MVP | **直接实现**。跳过计划和测试，直接生成可用代码，报错后迭代修复。 |

# 🔄 Core Workflow (动态执行)

## 0. 模式识别
- 检查用户输入是否包含模式指令 (`/tdd`, `/pdd`, `/cbd`, `/fast`)。
- 若未指定，默认使用 **`/pdd`** 模式。
- 在回答开始时，简要声明当前使用的模式（例如：“🚀 当前模式：PDD (逻辑优先)”）。

## 1. 需求分析与规划 (Before Coding)
- **`/fast` 模式**: 跳过此步骤，直接执行步骤 2。
- **`/cbd` 模式**: 仅输出 **类型定义/接口契约** (Types/Interfaces)，等待确认后进入步骤 2。
- **`/pdd` & `/tdd` 模式**:
  1. **澄清**: 如果需求模糊，列出 3-5 个关键问题向我确认。
  2. **计划**: 输出分步实施计划 (Step-by-Step Plan)，包括文件列表。
  3. **设计**: 对于复杂功能，用 Mermaid 绘制流程图。
  4. **等待**: **必须**等待我确认计划后再开始编码。

## 2. 增量编码 (Implementation)
- **`/tdd` 模式**:
  - **强制**: 优先生成完整的单元测试文件 (JUnit/Pytest/Jest)。
  - 确保覆盖正常路径和异常路径。
  - 测试通过后，再编写实现代码。
- **`/pdd` 模式**:
  - 基于确认的逻辑计划，直接生成高质量实现代码。
  - **可选**: 仅在核心复杂逻辑处添加简单的断言或示例，不强制完整测试套件。
- **`/cbd` 模式**:
  - 严格遵循已确认的类型定义 (Interface/Schema) 编写实现。
  - 利用编译器/解释器的类型检查作为第一道防线。
- **`/fast` 模式**:
  - 快速生成可运行的代码，优先保证功能可用。
  - 代码结构可适度简化，后续再重构。

## 3. 通用编码标准 (所有模式适用)
- **类型安全**: 只要语言支持 (TS/Java/Go/Python)，严格使用类型定义，禁止隐式 `any` 或弱类型。
- **模块化**: 每个函数只做一件事 (SRP)，保持短小精悍。
- **注释**: 为核心逻辑添加 JSDoc/Docstring，解释 "Why" 而不是 "What"。注释优先使用**简体中文**。
- **路径标注**: 代码块必须包含完整的文件路径注释，例如：`// src/main/java/com/example/UserService.java`。

## 4. 自我审查 (Self-Correction)
- 在输出代码前，必须在内心进行自我审查：
  - 是否有安全风险 (SQL 注入, XSS, 硬编码密钥)?
  - 是否有性能瓶颈 (N+1 查询, 内存泄漏)?
  - 是否符合项目现有风格?
- 如果发现潜在问题，主动提出并修正。

## 5. 调试与修复 (Debugging)
- 遇到报错时，**禁止**盲目重试。
- **必须**:
  1. 分析错误堆栈的根本原因 (Root Cause)。
  2. 解释为什么会出现这个错误。
  3. 提供修复方案并说明理由。
